XMLEncoder and XMLDecoder

基础知识

XMLDecoder/XMLEncoder 并不是 Weblogic 特有的， 它是在 JDK 1.4版添加的 XML 格式序列化持久性方案。因此他其实是属于 Java 的。

关注两个点：

  1. XML 格式：前置知识：XMLDecoder语法分析
  2. 序列化：个人理解其实序列化， 反序列化 都是为了方便数据的传输等等。无论是什么语言，无非是通过函数将对象存储成字符串也好，二进制文件也好，json也好，xml也好，最终就只是一个载体，然后通过反序列化相关函数将载体读取，进行对象的还原。

相关知识

在Java中解析XML一般有两种方式，DOM和SAX，前者是一次性读取Xml内容存入内存，便于操作，后者是顺序读取内容，遇到什么标签再根据自己的定义操作去解析。XmlDecoder是基于Java自带SAX Xml进行解析的。

JDK 1.6中XMLDecoder的主要解析逻辑都在 com.sun.beans.ObjectHandler 类中 。
JDK1.7以后XMLDecoder的解析逻辑进行了重构，解析逻辑打散在了 com.sun.beans 包中的20多个类中。

DocumentHandler 的构造函数主要作用是创建各个标签对应的 ElementHandler

DefaultHandler 主要靠 ContentHandler 接口解析 XML 。相关内容 —ContentHandler的文档

@fnmsd 小结如下：

startElement
处理开始标签，包括属性的添加

  • DocumentHandler: XML 解析处理过程中参数包含命名空间URL、标签名、完整标签名、属性列表。根据完整标签名创建对应的ElementHandler并添加相关属性，继续调用其startElement。
  • ElementHandler: 除了array标签以外，都无操作。

endElement
结束标签处理函数

  • DocumentHandler: 调用对应ElementHandler的endElement函数，并将当前ElementHandler回溯到上一级的ElementHandler。
  • ElementHandler: 没看有重写的，都是调用抽象类ElementHandler的endElement函数，判断是否需要向parent写入参数和是否需要注册标签对象ID。

characters

  • DocumentHandler: 标签包裹的文本内容处理函数，比如处理java.lang.ProcessBuilder包裹的文本内容就会从这个函数走。函数中最终调用了对应ElementHandler的addCharacter函数。
  • addCharacter
  • ElementHandler: ElementHandler里的addCharacter只接受接种空白字符(空格\n\t\r)，其余的会抛异常，而StringElementHandler中则进行了重写，会记录完整的字符串值。

addAttribute

  • ElementHandler: 添加属性，每种标签支持的相应的属性，出现其余属性会报错。

getContextBean

  • ElementHandler: 获取操作对象，比如method标签在执行方法时，要从获取上级object/void/new标签Handler所创建的对象。该方法一般会触发上一级的getValueObject方法。

getValueObject

  • ElementHandler: 获取当前标签所产生的对象对应的ValueObject实例。具体实现需要看每个ElementHandler类。

  • isArgument
    ElementHandler: 判断是否为上一级标签Handler的参数。

  • addArgument
    ElementHandler: 为当前级标签Handler添加参数。

XMLDecoder反序列化的流程中两个字段
两个成员变量，在类的实例化之前，通过对parent的调用进行增加参数。

  1. parent
    最外层标签的ElementHandler的parent为null，而后依次为上一级标签对应的ElementHandler。
  2. owner
    ElementHandler: 固定owner为所属DocumentHandler对象。
    DocumentHandler: owner固定为所属XMLDecoder对象。

CVE-2017-3506

无过滤的情况

几个关键点如下：

解析 <object class="java.lang.ProcessBuilder"> 时 ObjectElementHandler#addAttribute 中没有相应的匹配 class name ，这里会调用父类 NewElementHandler#addAttribute

使用 findclass 加载 java.lang.ProcessBuilder 并附值给 type

对应着解析 <array class="java.lang.String" length="3"> 时 ArrayElementHandler#addAttribute 也没有相应的匹配 class name ,这里也会调用父类 NewElementHandler#addAttribute （继承关系图）

在跟 void 标签解析的时候发现了有趣的现象。this.handler.addAttribute(var7, var8); 会直接跳到 ObjectElementHandler#addAttribute 去解析。原因可以参考 @p1g3 这一段话

void标签对应的VoidElementHandler其实就是一个空壳，当我们调用这个handler的方法时，由于继承的关系，除了isArgument的无参形式会调用VoidElementHandler#isArgument外，其它的其实都会调用ObjectElementHandler的对应方法

后面依次解析，在退出 void 之后 进入了 ElementHandler#endElement

获取对象依次触发：ElementHandler#endElement -> NewElementHandler#getValueObject -> ObjectElementHandler#getValueObject ->NewElementHandler#getContextBean 。

在这时 this.type == null 因此调用父类 ElementHandler#getContextBean

后续调用 NewElementHandler#getValueObject -> ObjectElementHandler#getValueObject。这里调用 this.getContextBean 因为自己有 type 了所以 直接返回自己的 return this.type != null ? this.type : super.getContextBean();

最终通过 Expression 创建对象

后面调用方法同理通过Expression调用了 start 函数
