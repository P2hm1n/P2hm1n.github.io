序列化与反序列化

正常情况下的序列化与反序列化，摘自 @Hu3sky 师傅

正常情况下的序列化

  1. server 将远程代理对象bind到注册中心。
  2. Clinet将字符串（要调用的类的别名）序列化传给注册中心
  3. 注册中心返回远程代理对象给Client。
  4. clinet 将 调用方法的参数写到 server 端,marshalValue方法，本例是一个string类型。（代理对象间）
  5. 如果方法不是void类型，Server端将执行结果返回给client（代理对象间）。

正常情况下的反序列化

  1. 注册中心反序列化来自Server端的远程对象，一次是String类型，一次是Remote类型。
  2. 注册中心反序列化Client端的String（要调用的类的别名）。
  3. client 读取来自注册中心返回的代理对象。
  4. Server端调用unmarshalValue，获取来自Client的传参。（代理对象间）
  5. client 调用unmarshalValue方法 读取 server 端方法的 return 值。（代理对象间）

攻击注册中心

攻击原理

攻击注册中心时，攻击面来源于不同 case 中对序列化对象的处理（反序列化）

因此可以根据其特征分为三类

  • 没有 readObject：list
  • 有 readObject，同时具有两种类型的 readObject：bind、rebind
  • 有 readObject，只有 String 类型：lookup、unbind

关于 bind/rebind 的限制， JDK 8u141 版本上下的差异

首先看 <JDK8u141，以 bind 方法举例

跟进 bind 方法

可以看出具体的流程是： 先获得对象 -> 反序列化 -> checkAccess

但是 check 前我们就已经可以触发反序列化发起攻击了

再看看 >JDK8u141

rebind 具体情况同 bind 方法一样

这样的情况带来的问题是：如果要使用 bind/rebind 方法来远程攻击Registry，JDK版本必须在8u141之前

而 readObject 带来的安全隐患就是如果 Registry 端有什么有漏洞的组件，那么可以直接打，如 CC

关于 lookup/unbind 的问题

以 lookup 为例，因为 lookup 可以解决 JDK版本在8u141之后的远程攻击问题

但 lookup 虽然没有限定请求来源，但是他缺少了对远程对象的 readObject，只允许字符串类型

解决方法是在构造请求的时候重新实现 lookup 函数就行了
